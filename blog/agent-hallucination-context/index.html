My Business Agent Drafted a Fake Client Plan. Here’s Exactly Why That Happened.

Jett Nguyen · June 30, 2026 · 5 min read

I was testing the business agent I built for Otian on something real: drafting an email with an agenda and action items for a client. The agent came back almost immediately. Professional tone, clear structure, specific-sounding deliverables. It looked great.

It was also completely made up.

The action plan it wrote wasn’t for that client. It wasn’t even close. The agent had invented a generic, plausible-sounding plan and presented it with total confidence, as if it had known the client for months.

What Actually Happened

An AI agent doesn’t store knowledge the way you do. It doesn’t remember your clients between sessions or carry context in its head. Every time an agent runs, it starts from scratch. What it knows depends entirely on what information it loads at the start.

Those startup files are called bootstrap files. Think of them as the agent’s briefing document: who the client is, what the project involves, what’s already been decided, what’s still open. Before the agent does anything, it reads those files and builds a working picture of the situation.

The routing is what tells the agent which files to load and when.

On that test run, the routing had a gap. The bootstrap files for that client either weren’t loaded or were skipped entirely. So when the agent went to write the email, it had nothing real to work from: no client name, no project context, no actual action items. Just a prompt asking it to draft something professional.

So it did what language models do when they don’t have what they need: they fill in the gaps. Confidently.

The agent wasn’t lying. It genuinely didn’t know it didn’t know.

The Confidence Is the Dangerous Part

This is what makes hallucination actually tricky. If the agent had said “I don’t have enough information to write this,” you’d fix the problem. But it didn’t. It produced something that looked completely real: professional formatting, specific language, the right structure. Nothing in the output signaled that any of it was invented.

That’s the nature of how these models work. They don’t experience uncertainty the way you do. They reason from what’s in front of them, and if what’s in front of them is incomplete, they complete the picture themselves. The output quality doesn’t drop. The confidence doesn’t waver. It just makes something up.

What This Taught Me About Building Agents

After that draft, I went back and tightened the routing. Every client context, every project file, every relevant document loads before the agent touches anything else. The agent doesn’t get to write a single word until it’s been properly briefed.

But the deeper lesson was this: the decision the agent made wasn’t wrong. Given what it had, it did exactly what it was supposed to do: write a professional email with an agenda and action items. The inputs were broken. The decision was technically correct.

Building a good agent is mostly about getting the inputs right. The model will handle the rest, for better or worse.

Most people think about agents in terms of outputs: did it do what I asked? The real question is: did it have what it needed to do it correctly? That’s the part you build. That’s the part that determines whether you get a real plan or a convincing fiction.

This Is Why We Don’t Set It and Forget It

At Otian, we don’t hand someone an agent and walk away. We build the context architecture: the bootstrap files, the routing logic, the information the agent has access to. We keep refining it as the picture of your work gets clearer. An agent that knows everything it needs to know is almost never wrong. An agent running on incomplete context is almost always confidently wrong.

The difference isn’t the model. It’s the setup.

If you’ve ever tried an AI tool that seemed to just make things up, that’s almost certainly what happened. And it’s fixable, but you have to know what to fix.
