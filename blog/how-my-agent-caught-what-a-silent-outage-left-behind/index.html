How My Agent Caught What a Silent Outage Left Behind

Jack Raney · July 9, 2026 · 9 min read

It started with a complaint, not a crisis. I told my agent that the skill which tracks email follow-ups for me hadn't fired. Not "hasn't fired recently." Not once, ever, as far as I could remember. No urgency to it — more the mild annoyance of a feature you assume is running quietly in the background until you notice it never actually announced anything.

I expected a quick answer. I got a two-hour investigation that turned up three separate bugs, one of which would have sent me to the wrong place at the wrong hour for a standing personal appointment.

The Feature That Never Ran

The first problem was figuring out what "the follow-up skill" even was. It isn't a single toggle — it's a small daemon (a background process that just keeps running, checking your inbox on a timer, the way a thermostat keeps checking the temperature) that reads new email and tries to infer commitments: promises, asks, things a reasonable person would follow up on. When it finds one, it's supposed to surface it.

My agent didn't take my framing at face value. Instead of assuming the feature was misconfigured, it went looking for where the feature actually lived, traced it to the daemon, and checked its output directly: a store of "inferred follow-up commitments," queried with a single command.

The count was zero. Not "zero pending" — zero, total, ever. Nothing had ever been written to it.

That's the kind of result that tells you the problem isn't the feature's logic. If the logic were wrong, you'd expect some garbage in there — bad inferences, duplicates, something. Zero, forever, means the feature never got a single email to look at in the first place.

Two Thousand, Four Hundred, and Ninety-Seven Identical Lines

The daemon's log confirmed it. Every sixty seconds, it tried to check my inbox. Every sixty seconds, for what turned out to be days, it failed with the exact same error: invalid_grant: Token has been expired or revoked.

Gmail access for an automated process works through OAuth (the login handshake that lets an app act on your behalf without ever seeing your password) and a refresh token — a long-lived credential that lets the process quietly renew its short-term access every hour without bothering you. That part had been working fine for weeks; the daemon refreshed itself automatically and I never noticed. What died wasn't the hourly refresh. It was the refresh token itself — the thing underneath the thing that renews. Once that's gone, no amount of automatic retrying fixes it. You need a human to sit through a login flow again.

Nobody had. There was no error to click through, no notification, nothing. Just a log file, growing quietly, that nobody was reading.

My agent ran the re-authorization flow, got a fresh token, and confirmed — by hand, hitting the Gmail API directly — that it worked.

And the daemon still didn't recover.

The Bug Behind the Bug

This is the part I'd have stopped at, if I'd been doing this myself. Token's fixed, ticket's closed, move on. My agent didn't move on, because the daemon's own log had gone completely silent — no errors, but also no successes. A process that's supposed to check something every sixty seconds should be saying something every sixty seconds, even if that something is "nothing new."

It checked the process directly: still running, using zero CPU, doing nothing. Not crashed — worse, in a way. A crashed process at least tells you it's dead. This one just sat there, holding a working credential it had stopped trying to use, hung on some earlier connection that never came back.

The fix was almost anticlimactic: restart it. Once it came back up with a clean connection and the valid token, it immediately found something real — a missed reminder from days earlier — and handled it correctly within seconds.

Two separate failures, stacked on top of each other, and the second one was only visible because nobody stopped at "the token's fixed now."

Why Does This Keep Happening?

I asked the obvious next question: why did the credential die in the first place, and how do I stop it happening again?

The honest answer is I still don't have full certainty — that requires checking a setting in Google's developer console that isn't reachable from a terminal. But the investigation surfaced something I hadn't known: there were two different sets of credentials sitting in the same config directory, created weeks apart, for the same project. Which means this had already happened once before. Someone's response to it back then — me, presumably, or an earlier version of this same automation — wasn't to diagnose the failure. It was to just generate a new credential and move on, the way you'd get a new key cut instead of asking why the old lock keeps jamming.

That's the real finding, more than the specific OAuth theory. A recurring failure that gets patched instead of understood just goes quiet again. So instead of only fixing the token, my agent added something that hadn't existed before: if the daemon fails five times in a row — a few minutes of real trouble, not a network blip — it now sends me an actual alert. Not a line in a log file nobody tails. A message. And when it recovers, it tells me that too. The next time this credential dies, however it dies, I'll know within five minutes instead of finding out weeks later by noticing a suspiciously empty list.

The Appointment That Would Have Been Two Hours Wrong

While all this was still being sorted out, a message came through from the same system, working exactly as intended: a recurring personal appointment detected from an email, a calendar event proposed automatically, a reminder set for the drive.

Appointment detected — [contact] — Jul 9 at 7:00 PM. In-person (50 min).

Something about it read wrong. Not obviously wrong — just a nagging sense that the time didn't match what I expected. I asked my agent to check.

The actual calendar event, it turned out, was correct: 5:00 PM, in the time zone that matched where the appointment actually was. The message describing it said 7:00 PM. Same appointment, same underlying data, two different times, two hours apart — because the code that generates the human-readable text had been hardcoded to describe every appointment in my home time zone, regardless of where the appointment actually was. When an earlier version of the automation added a special case for "if I'm physically somewhere else, book this in person instead of over video," someone updated the calendar event to use the correct time zone and simply forgot to update the sentence describing it to match.

The event itself was never wrong. If I'd tapped confirm, my calendar would have shown the correct time. But I don't check the invite twice — I read the message, I show up. If that message had gone unquestioned, I'd have shown up two hours off from the actual appointment time.

The instructive part isn't the two-hour gap. It's why it existed: two pieces of information that should never be able to disagree were being computed by two completely separate lines of code, in two different places, written at two different times. Nothing was stopping them from drifting apart — and eventually, quietly, they had.

The fix wasn't just correcting the displayed time. It was removing the possibility of this happening again: instead of the message and the calendar event each doing their own time zone math, both now pull from a single shared calculation. There is no longer a version of this bug where they can disagree, because there's only one place left where the time gets decided at all.

The Detail I Only Noticed Because I Was Already Looking

With the time zone fixed, I asked one more question, mostly out of curiosity: did this appointment even follow the format I'd specified for how my calendar titles should look?

It didn't. I have a documented standard for exactly how these should read — a specific word order for how a contact's name and the venue relate to each other — and the automated version had it backwards. Not broken, not confusing, just not what I'd actually specified. The kind of small inconsistency that's easy to live with indefinitely because it never causes a visible problem, only a quiet gap between what you asked for and what you're getting.

That one wasn't hiding behind an error message. It was hiding behind the fact that nobody had ever put the actual rule next to the actual code and compared them line by line. Once that comparison happened, the mismatch took seconds to find and seconds to fix.

Trying to Break It on Purpose

By this point I'd found three unrelated bugs while chasing one complaint about a missing feature. That pattern — a small question keeps leading somewhere else — is usually a sign there's more to find, not less. So rather than call it done, I asked for a stress test: don't just tell me it's fixed, prove it.

My agent built a harness that imported the actual code and ran it against a battery of fabricated scenarios — different date formats real emails might use, a version of the appointment logic run with location flipped both ways, a duplicate reminder for an appointment that already exists, a deliberately garbled email with no usable date in it at all — all without sending a single real message or touching a single real file. Forty-three checks in total.

Two failed on the first run. One was a flaw in the test itself, not the code — a shortcut that reset state between calls in a way the real daemon never does, quickly fixed by testing it the same continuous way it actually runs. The other wasn't a flaw in the test. It was a fourth bug: a date format the code claimed to support, using its own documented example as the test case, that had never actually been parseable. A missing comma, silently swallowed by an error handler that was written to fail quietly rather than loudly. There's no way to know how long that path had been dead, because nothing had ever needed it to notice.

Forty-three checks, four real bugs found across the whole session, and every one of them was invisible until something insisted on being wrong out loud instead of failing silently.

What Actually Happened Here

None of these bugs were dramatic. Nothing was hacked, nothing was stolen, nobody was locked out of anything. The worst-case outcome, if none of this had surfaced, was closer to quietly annoying than dangerous: a feature that never did anything, an appointment I might have shown up to two hours off, a calendar that didn't quite match my own stated preferences, and a code path that had apparently never worked and never would have been missed.

But that's exactly the shape most real problems take. They don't announce themselves. They sit in a log file nobody's tailing, or a display string that quietly stopped matching the thing it was describing, or an error handler that was written to fail politely instead of loudly. Nobody sits down and audits their automation for its own sake. You notice when something's a little off, you ask, and then the question either stops at the first plausible answer or it doesn't.

What mattered here wasn't any single fix. It was that the investigation didn't stop once the original complaint was technically resolved. Token fixed, but the process is still silent — check why. Time is fixed, but does it match the format I asked for — check that too. Everything looks fine now — prove it, instead of assuming it.
